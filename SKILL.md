# 웹시스템 사용자 매뉴얼 생성

웹 시스템을 분석해 최종 사용자용 매뉴얼을 생성하는 워크플로. 최종 산출물은
`사용자매뉴얼_<시스템명>_<YYYYMMDD>.docx` 및/또는 `.pptx`이며, 모든 중간 산출물은
대상 프로젝트의 `manual-work/` 아래에서 관리한다.

## 0. 동작 모드

시작 시 아래 세 모드 중 하나를 판정하고 사용자에게 알린다. 이후 단계는 모드에 따라 분기된다.

| 모드 | 조건 | 특징 |
|---|---|---|
| A. 하이브리드 | 소스코드 + 실행 URL 모두 확보 | 기본이자 최고 품질. 소스로 화면 목록을 잡고 라이브로 검증·캡처 |
| B. 소스 온리 | 실행 시스템 없음 | 스크린샷은 placeholder 처리, 코드로 확정 불가한 항목은 `[확인 필요]` 마킹 |
| C. 라이브 온리 | 소스 접근 불가 | 브라우저 순회만으로 화면 목록 구축 |

## 1. 시작 전 점검과 인터뷰

### 자동 감지 (사용자에게 묻지 않는다)

질문을 줄일수록 작업 시작이 빨라진다. 다음은 스스로 확인한다:

1. **재개 여부**: `manual-work/inventory.md`가 이미 있으면 이전 작업이 중단된 것이다.
   config.md와 inventory.md를 읽고 진행 상황을 요약한 뒤 "이어서 진행할지"를 먼저 제안한다.
   재개 시 인터뷰를 반복하지 않는다.
2. **소스 경로**: 현재 작업 디렉토리에 웹 프로젝트 마커(pom.xml, build.gradle, package.json 등)가
   있으면 그것이 대상이다. 없으면 인터뷰에서 경로를 묻는다.
3. **기술 스택**: `references/source-analysis.md`의 감지 표를 적용한다.
4. **캡처 수단·브라우저**: `references/live-capture.md`의 의사결정 트리 ①~④ 순서로
   가용성을 판별한다. (① claude-in-chrome MCP → ② 디버그 포트+CDP → ③ headless → ④ OS 창 캡처)
   캡처 브라우저는 기본 브라우저 설정과 무관하게 **Chrome 우선, 없으면 Edge**로 고정하고
   config.md에 기록한다 — 스크린샷이 한 브라우저로 통일되어야 하기 때문이다.

### 한 번에 묶어서 질문한다

흩어진 질문은 사용자를 지치게 하므로 아래를 한 번에 묻는다(AskUserQuestion 권장):

1. 실행 중인 시스템 URL 유무. 있다면 로그인 필요 여부.
2. 매뉴얼 대상 독자(최종 사용자 / 시스템 관리자 / 둘 다)와 범위(전체 메뉴 / 특정 메뉴).
3. 출력 형식: docx / pptx / 둘 다. pptx 선택 시 서식을 복제할 참고 템플릿 pptx 유무.
4. 시스템 공식 명칭과 버전(표지 표기용).
5. 화면에 운영 데이터(실명·연락처 등)가 보여도 되는지 — 테스트 환경/계정 유무.

### 로그인 계정 처리 원칙 (무개입 우선)

비밀번호를 채팅으로 입력받지 않는다 — 대화 기록에 자격 증명이 남기 때문이다.
로그인은 개입이 적은 순서로 시도한다:

1. **세션 재사용(무개입)**: CDP 경로(②)는 고정 캡처 프로필을 쓰므로 이전 실행의 로그인
   쿠키가 남아 있다. cdp_capture.py가 로그인 상태를 자동 판정해 유효하면 그대로 진행한다.
   MCP 경로(①)도 사용자의 실제 브라우저 세션을 그대로 재사용한다.
2. **환경변수 자동 로그인(무개입)**: 세션이 만료됐고 환경변수 `WUM_LOGIN_ID`/`WUM_LOGIN_PW`가
   설정돼 있으면 cdp_capture.py가 직접 로그인 폼을 채운다. 자격 증명은 스크립트만 읽으며
   대화에 노출되지 않는다 (설정 방법은 live-capture.md).
3. **직접 로그인 요청(폴백)**: 1·2가 모두 불가하면(환경변수 없음, SSO/2FA, 셀렉터 불일치 —
   cdp_capture.py 종료 코드 3) 사용자에게 캡처 프로필 Chrome 창(또는 MCP 브라우저)에서
   직접 로그인해 달라고 요청한 뒤 이어서 진행한다.

테스트 계정이 문서에 실려야 하는 경우(예: 매뉴얼의 로그인 절차 예시)에만 계정 "아이디"를 받는다.

인터뷰 결과는 즉시 `manual-work/config.md`에 기록한다 — 재개 시 재질문을 막기 위함이다.

## 2. 작업 공간 규약

대상 프로젝트 루트에 아래 구조를 만든다:

```
manual-work/
├── config.md          # 인터뷰 결과: 시스템명, URL, 독자, 범위, 출력 형식, 캡처 수단
├── inventory.md       # 화면 인벤토리 = 진행 상황 원장 (아래 포맷)
├── screenshots/       # SCR-<ID>_<영문슬러그>.png (+ _annotated.png)
├── sections/          # 화면/메뉴 단위 마크다운 조각 (예: 10-user-mgmt.md)
└── manual-draft.md    # 병합된 최종 초안
```

`inventory.md` 포맷 — 이 표가 곧 체크리스트이자 재개 지점이다:

```markdown
| ID      | 메뉴 경로               | 화면명       | URL/라우트     | 소스 근거              | 캡처 | 문서화 |
|---------|-------------------------|--------------|----------------|------------------------|------|--------|
| SCR-001 | 시스템관리 > 사용자관리  | 사용자 목록  | /admin/users   | UserController.java:45 | 완료 | 완료   |
| SCR-002 | 시스템관리 > 권한관리    | 권한 설정    | /admin/roles   | routes.tsx:88          | 실패(권한없음) | 대기 |
```

- 상태 값: 캡처 = `대기 | 완료 | 실패(사유) | 불필요`, 문서화 = `대기 | 진행중 | 완료`
- **상태가 바뀔 때마다 inventory.md를 즉시 갱신한다.** 세션이 끊겨도 이 파일만으로
  이어서 작업할 수 있어야 하기 때문이다.
- 스크린샷에 운영 데이터가 섞일 수 있으므로, 대상 프로젝트가 git 저장소면
  `manual-work/`를 `.gitignore`에 추가하도록 제안한다.

## 3. 화면 인벤토리 구축 (소스 분석 — A/B 모드)

이 단계를 시작할 때 `references/source-analysis.md`를 읽는다 (스택별 추출 위치와 grep 패턴).

1. 마커 파일로 스택을 감지하고 config.md에 기록한다.
2. 해당 스택 섹션의 힌트로 라우트/컨트롤러/메뉴 정의/뷰 파일에서 화면 후보를 추출한다.
3. **화면 vs API를 구분한다**: JSON만 반환하는 엔드포인트는 제외한다 — 매뉴얼은
   사용자가 보는 화면 단위이기 때문이다.
4. 메뉴 라벨은 우선순위대로 확보한다: 메뉴 설정/컴포넌트 → i18n 리소스 → DB 메뉴 테이블
   시드 SQL → 라이브 실측(A모드) → `[확인 필요]`(B모드).
5. 목록/상세/등록/수정 CRUD 화면군은 하나의 기능 단위로 묶는다 — 독자는 화면 4개가 아니라
   "사용자 관리 업무" 하나로 인식하기 때문이다.
6. 결과를 inventory.md로 작성하고 **사용자에게 범위 확인을 받은 뒤** 다음 단계로 간다.
   화면이 50개를 넘으면 메뉴 단위로 나눠 여러 회차로 진행할 것을 제안한다.

C 모드에서는 이 단계를 건너뛰고 4단계의 메뉴 실측으로 인벤토리를 만든다.

## 4. 라이브 순회와 스크린샷 (A/C 모드)

이 단계를 시작할 때 `references/live-capture.md`를 읽는다 (캡처 수단별 절차, 배지 합성, 안전 수칙).

1. **접속·로그인**: config.md에 기록된 캡처 수단으로 URL에 접속한다. 로그인은 1단계 원칙대로.
2. **메뉴 실측 대조**: 페이지의 GNB/LNB 메뉴 트리를 읽어 inventory와 대조한다.
   소스에 없던 화면은 추가하고, 권한 등으로 접근 불가한 화면은 상태를 기록한다.
3. **화면별 순회 루프** — 5화면 내외 배치로 "이동 → 캡처 → 기록 → inventory 갱신"을 반복한다.
   배치 단위로 끊는 이유는 컨텍스트가 길어져도 손실이 배치 하나로 국한되게 하기 위함이다:
   - 화면으로 이동해 스크린샷으로 상태를 확인한다 (로딩 완료, 에러 없음).
   - 캡처 파일을 `screenshots/SCR-<ID>_<슬러그>.png`로 확보한다 (수단별 방법은 live-capture.md).
   - 화면 구성을 관찰 즉시 `sections/`의 해당 파일에 기록한다: 버튼 목록, 입력 항목과
     필수 여부, 표 컬럼, 검색 조건, 페이징, 상태 전이.
   - 번호 배지가 필요한 화면은 요소별 상대 좌표를 정해 `scripts/annotate_screenshot.py`로
     합성한다. 원본은 보존되고 `_annotated.png`가 생긴다.
4. **읽기 전용 원칙**: 조회·검색·탭 전환·페이징·모달 열기까지만 허용한다.
   **저장/등록/수정/삭제/제출 버튼은 클릭하지 않는다** — 운영 데이터를 변경할 수 있기 때문이다.
   등록 화면의 캡처는 열기만 하고 제출 없이 담는다. 유효성 메시지 유도(빈 제출)는
   사용자가 테스트 환경임을 확인해 준 경우에만 한다.
5. **개인정보**: 실명·연락처·주민번호 등이 화면에 보이면 캡처 전에 사용자에게 알리고
   진행 여부를 확인한다. 필요 시 live-capture.md의 블러 처리 방법을 쓴다.

## 5. 화면별 문서화와 매뉴얼 조립

이 단계를 시작할 때 `references/manual-template.md`를 읽는다 (표준 목차, 화면 포맷, 표기 규약).

- 문체: 최종 사용자 대상 존댓말("~합니다"). 독자는 코드를 모르는 사람이므로 기술 용어를
  쓰지 않는다 — "API를 호출합니다"가 아니라 "[저장]을 클릭하면 반영됩니다".
- 화면에 보이는 라벨을 그대로 인용하고, 버튼은 `[버튼명]` 대괄호로 표기한다.
- 화면 단위 포맷(개요 → 접근 경로 → 스크린샷 → 번호별 기능 설명 → ※ 주의사항)은
  manual-template.md의 규약을 따른다.
- 스크린샷이 없는 화면은 placeholder 규약을 쓴다:
  `> [스크린샷 필요: SCR-012 사용자 등록 화면 — /admin/users/new]`
  문서 변환 시 회색 안내 상자로 렌더하고 inventory에 `대기`로 남긴다 — 나중에 라이브
  접근이 가능해지면 재실행으로 채울 수 있다.
- `sections/` 조각을 메뉴 트리 순서로 병합해 `manual-work/manual-draft.md`를 만들고,
  **사용자에게 목차 수준의 검토를 받은 뒤** 문서 생성으로 넘어간다.

## 6. 문서 생성 (docx / pptx)

이 단계를 시작할 때 `references/output-formats.md`를 읽는다 (형식별 변환 명세).

1. 사전 처리: `python scripts/resize_images.py manual-work/screenshots --max-width 1400`
   — 과대 이미지는 문서 용량과 생성 시간을 키우기 때문이다. (Pillow 미설치 환경이면 건너뛴다.)
2. **docx**: Skill 도구로 docx skill(`anthropic-skills:docx`)을 호출해 manual-draft.md와
   스크린샷으로 문서를 생성한다. 표지·자동 목차·헤딩 계층·그림 캡션·페이지 번호·맑은 고딕 등
   요구 명세는 output-formats.md의 "docx 변환 명세"를 따른다.
3. **pptx**: Skill 도구로 pptx skill(`anthropic-skills:pptx`)을 호출한다.
   - 사용자가 참고 템플릿 pptx를 제공했다면: 템플릿을 unpack해 기존 화면 슬라이드를 서식
     원본으로 복제하고 텍스트/이미지만 교체한다 — 조직의 기존 매뉴얼과 형식이 일치해야
     하기 때문이다.
   - 템플릿이 없다면: output-formats.md의 신규 레이아웃(표지 → CONTENTS → 챕터 간지 →
     화면 슬라이드)으로 생성한다.
   - 조직 표준 서식이 자주 쓰이면 이 skill의 `assets/`에 템플릿을 추가해 두고 재사용한다.
4. 폴백: docx/pptx skill을 호출할 수 없는 환경이면 manual-draft.md를 최종 산출물로
   확정하고 변환 방법(pandoc 등)을 안내한다. 강제로 도구를 설치하지 않는다.

## 7. 마무리 검수

산출 직후 아래를 확인하고 결과를 보고한다:

- [ ] 목차와 본문 헤딩이 일치하는가
- [ ] [사진 N]/[그림 N] 번호가 연속적인가
- [ ] placeholder가 남아 있으면 잔존 목록을 사용자에게 보고했는가
- [ ] inventory.md의 모든 항목이 `완료` 또는 `불필요`인가
- [ ] 산출 파일 경로를 사용자에게 안내했는가

미완료 항목(캡처 실패, placeholder)은 숨기지 말고 명시적으로 보고한다 — 다음 회차 재실행의
출발점이 되기 때문이다.
