# 웹시스템 사용자 매뉴얼 생성

웹 시스템을 분석해 최종 사용자용 매뉴얼을 생성하는 워크플로. 최종 산출물
`사용자매뉴얼_<시스템명>_<YYYYMMDD>.docx|pptx`는 **대상 프로젝트 루트**(사용자 지정 시
그 경로)에 저장한다 — `manual-work/`는 .gitignore 권장 대상이라 최종본을 그 안에 두면
납품물이 유실될 수 있기 때문이다. 중간 산출물만 `manual-work/` 아래에서 관리한다.

**슬래시 인자**: `/manual-gen [URL 또는 소스 경로] [자유 키워드...]`
— 인자로 받은 값(URL, 경로, "관리자"/"사용자", "docx"/"pptx" 등)은 해당 인터뷰 항목을
대체한다. 인자에 `auto`(또는 "자동", "질문 없이")가 있으면 아래 무정차 모드로 동작한다.

## 0. 동작 모드

시작 시 아래 세 모드 중 하나를 판정하고 사용자에게 알린다. 이후 단계는 모드에 따라 분기된다.

| 모드 | 조건 | 특징 |
|---|---|---|
| A. 하이브리드 | 소스코드 + 실행 URL 모두 확보 | 기본이자 최고 품질. 소스로 화면 목록을 잡고 라이브로 검증·캡처 |
| B. 소스 온리 | 실행 시스템 없음 | 스크린샷은 placeholder 처리, 코드로 확정 불가한 항목은 `[확인 필요]` 마킹 |
| C. 라이브 온리 | 소스 접근 불가 | 브라우저 순회만으로 화면 목록 구축 |

## 1. 시작 전 점검 — 최소 질문 원칙

원칙: **물어서 알 수 있는 것보다 추론으로 알 수 있는 것이 훨씬 많다.** 아래 순서로
최대한 스스로 채우고, 사용자 확인은 기본 1회로 압축한다.

### 1-1. 자동 감지·추론 (사용자에게 묻지 않는다)

0. **환경 점검**: `python scripts/check_env.py`를 실행해 의존성·브라우저·가능 경로를
   확인하고 스냅샷 한 줄을 config.md에 기록한다 — 환경마다 설치물이 달라 폴백이
   제각각 발동하면 산출 품질이 달라지므로, 결핍은 작업 후반이 아니라 시작 시점에
   드러나야 한다. 필요한 설치(python-pptx 등)는 이 시점에 사용자 확인 후 진행한다.
1. **설정 재사용**: `manual-work/config.md`가 있으면 그 값을 전부 기본값으로 쓴다
   (이전 실행의 인터뷰 결과이므로). inventory.md까지 있으면 재개를 제안한다.
   → 두 번째 실행부터는 사실상 질문이 없어진다.
2. **소스 경로**: 현재 작업 디렉토리의 웹 프로젝트 마커(pom.xml, build.gradle,
   package.json 등)로 판단.
3. **기술 스택**: `references/source-analysis.md`의 감지 표.
4. **캡처 수단·브라우저**: `references/live-capture.md` 0절·1절 — Chrome 최우선,
   Chrome 불가 시에만 설치 브라우저 탐지 후 사용자 선택. 확정값은 config.md에 기록.
5. **시스템 명칭**: 로그인/메인 페이지 `title` → README 제목 → pom.xml `name` /
   package.json `name` 순으로 추론.
6. **실행 URL**: 사용자가 주지 않았으면 — 실행 중 로컬 서버(설정 파일의 server.port,
   dev 스크립트의 포트, 흔한 개발 포트 8080/3000/5173 응답 여부)에서 추론.
7. **대상 독자·독자 영역**: 단일 독자를 가정하지 말고 먼저 **독자 영역을 감지**한다 —
   한 시스템이 시스템관리자/기관(운영)관리자/일반 사용자 화면을 나눠 갖는 경우가 많기
   때문이다. 신호: 라우트 프리픽스 분리(/admin, /system, /user 등), 롤 정의(enum·권한
   테이블), 권한별 레이아웃·사이드바 분리, 로그인 후 롤별 리다이렉트.
   - 감지된 각 영역은 **활성/흔적을 판별**한다 — 라우트가 존재해도 실기능이 없는
     스텁 영역(예: 로그인 수단이 없는 사용자 라우트, "준비 중" 페이지)을 매뉴얼
     대상으로 잡으면 안 되기 때문이다. 판별 신호는 source-analysis.md의
     "영역 활성/흔적 판별"을 따른다.
   - 활성 영역이 1개면 그 독자로 확정한다.
   - **활성 영역이 2개 이상이면** 영역 목록을 활성/흔적 상태·근거와 함께 config.md에
     기록하고 1-2 일괄 확인에서 "이번에 만들 영역(하나만 / 전부)"을 묻는다 — 흔적
     의심 영역도 근거와 함께 목록에 보여 사용자가 최종 판단하게 한다. 전부라면
     **영역별로 별도 매뉴얼을 회차 순차 생성**한다 — 한 문서에 합치지 않는다(권한이
     달라 메뉴·화면 자체가 다르므로, manual-template.md 1절의 분리 원칙).
     auto 모드는 활성 영역 전부를 순차 생성하고 흔적 의심 영역은 제외한다.
   - 제외한 영역(흔적 포함)과 일부만 만들고 끝낸 영역은 최종 보고에 근거와 함께
     명시한다 — 조용히 제외하면 그 영역 매뉴얼이 없다는 사실 자체가 잊히기 때문이다.
8. **운영 데이터 여부**: URL이 localhost/127.0.0.1/사설 IP + 개발 포트면 테스트 환경으로
   추정(개인정보 확인 생략 가능). 그 외 도메인은 운영 가능성이 있으므로 확인 대상.
9. **출력 형식·방향·테마**: config.md 이력 → 없으면 pptx 기본(docx 는 확인
   단계에서 선택). **pptx 라면 슬라이드 방향(세로 A4 기본 / 가로 16:9, 빌더
   `--orientation`)과 색 테마도 함께 결정한다**. 테마 선택지는 **기본 3종 + 기타 1개**
   가 전부다(특정 조직 서식을 별도 표준 선택지로 두지 않는다):
   - `--theme`: navy 네이비+블루(기본) / forest 딥그린+틸 / charcoal 차콜+오렌지
   - **기타: "참고 템플릿 pptx 지정(파일 경로+파일명 입력)"** — 기본 동작은
     **색 테마 추출**(`--theme-from <템플릿.pptx>`): 템플릿의 dark·accent 색만
     추출해 번들 레이아웃에 입힌다. 레이아웃·규격(테두리·타일·병합·캡션)은 번들
     그대로라 디자인이 깨지지 않는다. **템플릿 슬라이드 서식을 그대로 복제(unpack·
     슬라이드 복제)하는 것은 사용자가 "서식 그대로"를 명시 요구할 때만** 6단계 ②
     경로로 진행하며, 이때 verify_pptx 통과가 필수이고 레이아웃 편차 가능성을
     사전 고지한다.
   config.md 이력에 템플릿 경로가 있어도 **다음 회차에 자동 재사용하지 않는다** —
   기본값으로 제시만 하고 테마 3종과 함께 재확인한다(템플릿 결정이 조직 표준처럼
   굳는 것을 막기 위함). auto 모드는 pptx 세로+navy 를 채택하고 최종 보고에 명시한다.

### 1-2. 단일 일괄 확인 (기본 모드: 질문 1회)

추론 결과를 항목 표(시스템명 / 대상 소스·URL / 독자 / 범위 / 출력 형식·방향·테마
(pptx: 세로·가로, navy·forest·charcoal 또는 참고 템플릿 pptx 경로) / 캡처 수단)로
정리해 **AskUserQuestion 한 번**으로 "이대로 진행 / 수정할 항목"을 받는다.
추론에 실패했거나 확신이 낮은 항목만 질문 옵션에 노출하고, 나머지는 표로 보여 주기만
한다 — 사용자가 읽고 승인하는 비용이 항목별 문답보다 훨씬 싸기 때문이다.

일괄 확인이 끝난 시점에 config.md에 `(확인 필요)` 같은 미확정 값이 남아 있으면 안 된다 —
잔존 항목은 반드시 질문 옵션에 포함해 해소한다 (auto 모드는 `(추론)` 기본값을 채택하고
최종 보고에 명시). 미확정 값이 표지·본문까지 흘러가는 것을 인터뷰 단계에서 차단하기 위함이다.

### 1-3. 무정차(auto) 모드 (질문 0회)

사용자가 `auto`/"자동으로"/"질문 없이"를 지시하면:

- 1-2의 일괄 확인을 생략하고 추론 기본값으로 즉시 시작한다.
- 3단계의 인벤토리 범위 확인, 5단계의 목차 검토 게이트도 생략한다.
- **안전 관련 개입은 auto여도 유지한다**: 운영 환경으로 추정되는 시스템의 개인정보
  노출 화면(1-1의 8), 직접 로그인 폴백(아래 로그인 원칙의 3), 읽기 전용 원칙의 예외 행위.
- 추론으로 결정한 모든 항목을 config.md에 `(추론)` 표시와 함께 기록하고, 최종 보고에서
  그 목록을 명시한다 — 무정차의 대가는 사후 검수 가능성으로 지불해야 하기 때문이다.

### 로그인 계정 처리 원칙 (무개입 우선)

비밀번호를 채팅으로 입력받지 않는다 — 대화 기록에 자격 증명이 남기 때문이다.
로그인은 개입이 적은 순서로 시도한다:

1. **세션 재사용(무개입)**: CDP 경로(②)는 고정 캡처 프로필을 쓰므로 이전 실행의 로그인
   쿠키가 남아 있다. cdp_capture.py가 로그인 상태를 자동 판정해 유효하면 그대로 진행한다.
   MCP 경로(①)도 사용자의 실제 브라우저 세션을 그대로 재사용한다.
2. **환경변수 자동 로그인(무개입)**: 세션이 만료됐고 환경변수 `WUM_LOGIN_ID`/`WUM_LOGIN_PW`가
   설정돼 있으면 cdp_capture.py가 직접 로그인 폼을 채운다. 자격 증명은 스크립트만 읽으며
   대화에 노출되지 않는다 (설정 방법은 live-capture.md).
3. **직접 로그인 요청(폴백)**: 1·2가 모두 불가하면(환경변수 없음, SSO/2FA, 셀렉터 불일치 —
   cdp_capture.py 종료 코드 3) 사용자에게 캡처 프로필 Chrome 창(또는 MCP 브라우저)에서
   직접 로그인해 달라고 요청한 뒤 이어서 진행한다.

테스트 계정이 문서에 실려야 하는 경우(예: 매뉴얼의 로그인 절차 예시)에만 계정 "아이디"를 받는다.

인터뷰 결과는 즉시 `manual-work/config.md`에 기록한다 — 재개 시 재질문을 막기 위함이다.

## 2. 작업 공간 규약

대상 프로젝트 루트에 아래 구조를 만든다:

```
manual-work/
├── config.md          # 인터뷰 결과: 시스템명, URL, 독자, 범위, 출력 형식(pptx면 방향·테마 또는 참고 템플릿 경로), 캡처 수단
├── inventory.md       # 화면 인벤토리 = 진행 상황 원장 (아래 포맷)
├── screenshots/       # SCR-<ID>_<영문슬러그>.png (+ _annotated.png)
├── sections/          # 화면/메뉴 단위 마크다운 조각 (예: 10-user-mgmt.md)
└── manual-draft.md    # 병합된 최종 초안
```

`inventory.md` 포맷 — 이 표가 곧 체크리스트이자 재개 지점이다:

```markdown
| ID      | 메뉴 경로               | 화면명       | URL/라우트     | 소스 근거              | 캡처 | 문서화 |
|---------|-------------------------|--------------|----------------|------------------------|------|--------|
| SCR-001 | 시스템관리 > 사용자관리  | 사용자 목록  | /admin/users   | UserController.java:45 | 완료 | 완료   |
| SCR-002 | 시스템관리 > 권한관리    | 권한 설정    | /admin/roles   | routes.tsx:88          | 실패(권한없음) | 대기 |
```

- 상태 값: 캡처 = `대기 | 완료 | 실패(사유) | 불필요`, 문서화 = `대기 | 진행중 | 완료`
- **독자 영역이 여러 개면 영역 단위로 구획한다**: inventory.md를 영역별 최상위 섹션으로
  나누고, 원고도 영역별로 분리한다(`manual-draft-<영역슬러그>.md`, sections/ 파일명에
  영역 접두). SCR 번호도 영역별 대역(예: 시스템관리자 SCR-9xx)으로 나누면 재개·검수가
  영역 단위로 깔끔해진다. 산출물 파일명은 영역명 접두를 따른다(output-formats.md).
- **상태가 바뀔 때마다 inventory.md를 즉시 갱신한다.** 세션이 끊겨도 이 파일만으로
  이어서 작업할 수 있어야 하기 때문이다.
- 스크린샷에 운영 데이터가 섞일 수 있으므로, 대상 프로젝트가 git 저장소면
  `manual-work/`를 `.gitignore`에 추가하도록 제안한다.
- **번들 스크립트 우선 원칙**: 캡처·배지·블러·문서 생성은 skill의 `scripts/`를 사용한다.
  번들이 커버하지 못하는 요구는 즉석 자작 대신 번들 스크립트에 옵션을 추가하는 것을
  먼저 검토하고, 불가피하게 자작한 스크립트는 세션 임시 경로가 아닌 `manual-work/tools/`에
  저장하고 config.md에 사유를 기록한다 — 세션이 끝나면 임시 경로는 사라져 재현성을 잃기
  때문이다. 즉석 python 스크립트에도 UTF-8 stdout 재설정 헤더를 넣는다(Windows cp949 대응).
  **자작 도구로 캡처하더라도 배지·테두리 파이프라인은 번들로 유지한다** — 화면별로
  `cdp_capture.py --mark-only --mark "셀렉터;..."`로 좌표를 산출하고
  `annotate_screenshot.py --markers-file`로 합성한다. 캡처 도구를 바꿨다는 이유로
  배지가 통째로 빠지는 것(1·2차 실행에서 실제 발생)을 막기 위한 규칙이다.

## 3. 화면 인벤토리 구축 (소스 분석 — A/B 모드)

이 단계를 시작할 때 `references/source-analysis.md`를 읽는다 (스택별 추출 위치와 grep 패턴).

1. 마커 파일로 스택을 감지하고 config.md에 기록한다.
2. 해당 스택 섹션의 힌트로 라우트/컨트롤러/메뉴 정의/뷰 파일에서 화면 후보를 추출한다.
3. **화면 vs API를 구분한다**: JSON만 반환하는 엔드포인트는 제외한다 — 매뉴얼은
   사용자가 보는 화면 단위이기 때문이다.
4. 메뉴 라벨은 우선순위대로 확보한다: 메뉴 설정/컴포넌트 → i18n 리소스 → DB 메뉴 테이블
   시드 SQL → 라이브 실측(A모드) → `[확인 필요]`(B모드).
5. 목록/상세/등록/수정 CRUD 화면군은 하나의 기능 단위로 묶는다 — 독자는 화면 4개가 아니라
   "사용자 관리 업무" 하나로 인식하기 때문이다.
6. 결과를 inventory.md로 작성한다. 확인 게이트는 규모에 비례시킨다:
   - 15화면 이하: 요약만 보고하고 확인 없이 바로 진행한다 (작은 범위는 잘못돼도
     되돌리는 비용이 작다).
   - 15화면 초과: 사용자에게 범위 확인을 받는다. 단 auto 모드면 확인 없이 진행한다.
   - 50화면 초과: 메뉴 단위로 배치를 나눈다 — auto 모드면 나눠서 연속 진행, 아니면
     여러 회차 분할을 제안한다.

C 모드에서는 이 단계를 건너뛰고 4단계의 메뉴 실측으로 인벤토리를 만든다.

## 4. 라이브 순회와 스크린샷 (A/C 모드)

이 단계를 시작할 때 `references/live-capture.md`를 읽는다 (캡처 수단별 절차, 배지 합성, 안전 수칙).

1. **접속·로그인**: config.md에 기록된 캡처 수단으로 URL에 접속한다. 로그인은 1단계 원칙대로.
   로그인 직후 화면의 계정·권한 표시(프로필 영역·헤더 라벨)를 확인해 **config.md의 대상
   독자와 권한이 다르면 1회 경고하고 진행 여부를 확인한다**(auto 모드에서도 유지) —
   다른 권한의 계정으로 캡처하면 메뉴 구성·화면 상태가 독자의 실제 화면과 달라지기 때문이다.
   독자 영역을 전환해 다음 회차를 시작할 때는 **그 영역 권한의 계정으로 재로그인**하고
   같은 확인을 반복한다.
2. **메뉴 실측 대조**: 페이지의 GNB/LNB 메뉴 트리를 읽어 inventory와 대조한다.
   소스에 없던 화면은 추가하고, 권한 등으로 접근 불가한 화면은 상태를 기록한다.
3. **데이터 준비**: 캡처 품질은 화면에 보이는 데이터가 결정한다 — 워크플로형 화면
   (작성→검수→승인→발송 등)은 각 단계 상태의 데모 레코드를 먼저 만들어 그 상태에서
   캡처하고, 지표·목록이 0/빈 상태면 기간 필터를 넓히거나 데이터를 준비한 뒤 캡처를
   우선 시도한다 (불가하면 본문에 사유 주석). 상세 규칙은 live-capture.md의 "데모 데이터" 절.
4. **화면별 순회 루프** — 5화면 내외 배치로 "이동 → 캡처 → 기록 → inventory 갱신"을 반복한다.
   배치 단위로 끊는 이유는 컨텍스트가 길어져도 손실이 배치 하나로 국한되게 하기 위함이다:
   - 화면으로 이동해 스크린샷으로 상태를 확인한다 (로딩 완료, 에러 없음 — CDP 경로는
     cdp_capture.py가 에러 화면을 자동 감지해 종료 코드 4로 알린다).
   - 캡처 파일을 `screenshots/SCR-<ID>_<슬러그>.png`로 확보한다 (수단별 방법은 live-capture.md).
     긴 화면은 `--full-page` 또는 섹션 분할 캡처를 쓴다.
   - **배지 합성은 기본 동작이다**: 설명할 요소 3~7개의 CSS 셀렉터를 캡처 시
     `--mark`로 함께 지정해 좌표를 자동 산출하고(`.markers.json`),
     `scripts/annotate_screenshot.py --markers-file`로 합성한다. 배지 생략은 화면 단위
     예외이며 사유를 inventory에 기록한다 — 좌표를 육안으로 추정하던 비용 문제는
     자동 산출로 해소됐으므로 "전량 생략"은 허용되지 않는다.
   - **캡처 후 셀프 체크**: 본문에서 `[대괄호]`로 언급할 요소가 스크린샷 안에 완전히
     보이는지, 이미지 하단 경계에 반쯤 잘린 텍스트·카드가 없는지 확인한다. 잘렸으면
     풀페이지/분할 재캡처 — 본문이 설명하는 것을 독자가 화면에서 찾을 수 없으면
     매뉴얼이 아니기 때문이다.
   - 화면 구성을 관찰 즉시 `sections/`의 해당 파일에 기록한다: 버튼 목록, 입력 항목과
     필수 여부, 표 컬럼, 검색 조건, 페이징, 상태 전이.
5. **읽기 전용 원칙**: 조회·검색·탭 전환·페이징·모달 열기까지만 허용한다.
   **저장/등록/수정/삭제/제출 버튼은 클릭하지 않는다** — 운영 데이터를 변경할 수 있기 때문이다.
   등록 화면의 캡처는 열기만 하고 제출 없이 담는다. 유효성 메시지 유도(빈 제출)는
   사용자가 테스트 환경임을 확인해 준 경우에만 한다.
6. **개인정보·민감 콘텐츠**: 실명·연락처·주민번호 등이 화면에 보이면 캡처 전에 사용자에게
   알리고 진행 여부를 확인한다. 블러는 cdp_capture.py의 `--redact-file`/`--redact-email`로
   캡처 시점에 적용한다(방법은 live-capture.md). 공지 배너의 타 기관 정보·내부 운영
   정보·시한부 공지도 캡처 전 닫거나 블러 대상에 포함한다.

## 5. 화면별 문서화와 매뉴얼 조립

이 단계를 시작할 때 `references/manual-template.md`를 읽는다 (표준 목차, 화면 포맷, 표기 규약).

- 문체: 최종 사용자 대상 존댓말("~합니다"). 독자는 코드를 모르는 사람이므로 기술 용어를
  쓰지 않는다 — "API를 호출합니다"가 아니라 "[저장]을 클릭하면 반영됩니다".
- **표준 문형 우선**: 설명 문장은 자유 작문보다 manual-template.md 5절의 화면 유형별
  표준 문형에 화면 라벨을 끼워 넣는 방식을 우선한다 — 작성 주체(모델·세션)가 달라도
  서술이 균질해야 하기 때문이다. 기술 용어·명령형 침투는 원고 린터가 WARN으로 잡는다.
- 화면에 보이는 라벨을 그대로 인용하고, 버튼은 `[버튼명]` 대괄호로 표기한다.
- 화면 단위 포맷(개요 → 접근 경로 → 스크린샷 → 번호별 기능 설명 → ※ 주의사항)은
  manual-template.md의 규약을 따른다.
- 스크린샷이 없는 화면은 placeholder 규약을 쓴다:
  `> [스크린샷 필요: SCR-012 사용자 등록 화면 — /admin/users/new]`
  문서 변환 시 회색 안내 상자로 렌더하고 inventory에 `대기`로 남긴다 — 나중에 라이브
  접근이 가능해지면 재실행으로 채울 수 있다.
- `sections/` 조각을 메뉴 트리 순서로 병합해 `manual-work/manual-draft.md`를 만든다.
  기본 모드에서는 **목차 수준의 검토를 받은 뒤** 문서 생성으로 넘어가고, auto 모드에서는
  검토 없이 진행하되 최종 보고에 목차 전체를 실어 사후 검수를 대신한다.

## 6. 문서 생성 (docx / pptx)

이 단계를 시작할 때 `references/output-formats.md`를 읽는다 (형식별 변환 명세).

1. 사전 처리: `python scripts/resize_images.py manual-work/screenshots --max-width 1400`
   — 과대 이미지는 문서 용량과 생성 시간을 키우기 때문이다. (Pillow 미설치 환경이면 건너뛴다.)
2. 생성 경로 — **번들 빌더가 기본**이다 (환경마다 다른 skill 생태계가 산출 구조를
   좌우하지 못하게, 결정론적 생성기를 표준 경로로 둔다):
   - **① 번들 빌더(기본)**: `scripts/build_pptx.py` / `scripts/build_docx.py` 실행 —
     output-formats.md 규격(표지·CONTENTS·고정 프레임·이미지 테두리·6개 초과 분할·마크다운
     서식 변환·placeholder 규격)이 내장되어 있고, **원고 사전 검증(validate_draft)**과
     자체 검증까지 수행하므로 CLI/데스크톱 어디서든 동일 구조가 나온다.
     생성기를 즉석에서 자작하지 않는다.
   - **② 전용 skill(특수 경로)**: 사용자가 **참고 템플릿 pptx를 제공해 서식 복제**
     (unpack → 슬라이드 복제 → 내용 교체)가 필요한 경우에만 docx/pptx skill
     (`anthropic-skills:docx`/`pptx`)을 호출한다 — 번들 빌더가 못 하는 유일한 작업이기
     때문이다. 이때도 output-formats.md의 규격을 요구사항으로 전달한다.
     **② 경로(전용 skill·자작 템플릿 빌더 포함) 산출물은 반드시
     `python scripts/verify_pptx.py <pptx> --draft <원고>` 를 통과해야 한다** —
     번들 빌더의 내장 규격(테두리·비율 유지·폭 균일·캡션·분할)이 이 경로에서는
     자동 적용되지 않으므로, 산출물 파일 기반 기계 검증으로 우회를 차단한다.
     ERROR 는 수정 후 재생성이 원칙이다(실제로 자작 템플릿 빌더가 테두리 19건
     누락을 무경고 산출한 사례가 있다).
   - **③ 최후 폴백**: ①의 의존성(python-pptx/python-docx)을 설치할 수 없으면
     manual-draft.md를 최종 산출물로 확정하고 변환 방법(pandoc 등)을 안내한다.
3. 빌더가 원고 검증 ERROR로 중단되면 원고를 규약에 맞게 수정하는 것이 원칙이다 —
   `--skip-validate` 우회는 사용자가 지시한 경우에만 쓴다.
4. 요구 명세(헤딩 계층·캡션·페이지 번호 정책 등)는 output-formats.md를 따른다.
   조직 표준 서식이 자주 쓰이면 이 skill의 `assets/`에 템플릿을 추가해 두고 재사용한다.

## 7. 마무리 검수 — 기계 검증 우선

육안이 아니라 grep·카운트로 확인할 수 있는 것은 반드시 그렇게 확인한다
(육안 검수는 이번에 통과해도 다음에 재발하기 때문이다):

- [ ] **미확정 마킹 잔존**: config.md·manual-draft.md·산출물 텍스트 추출본에서
      `확인 필요`, `확정 전`, `TBD`, `미정` 패턴 grep — 잔존 시 목록을 보고한다
- [ ] **마크다운 잔재**: 산출물 텍스트 추출본에서 `**`, `![` grep (번들 빌더는 자체 검증 포함)
- [ ] **목차-본문 대조**: CONTENTS(또는 TOC)에 본문의 모든 절이 나열되고 번호가 실제와
      일치하는가 (pptx는 텍스트 추출로 대조)
- [ ] **형식 완결성**: 3장 각 절에 개요·접근 경로·스크린샷(또는 placeholder)이 모두
      있는가 — `접근 경로` 출현 횟수 = 절 개수인지 대조
- [ ] **배지 적용률**: `_annotated.png` 수 / 캡처 수 — 생략 화면은 inventory에 사유가
      있는지, 생략 비율이 과반이면 최종 보고에 사유와 함께 명시했는가
- [ ] [사진 N] 번호가 연속적인가, pptx는 슬라이드별 설명 항목이 6개 이하인가
- [ ] placeholder 잔존 목록을 보고했는가 — 그중 **공개 페이지(로그인·메인 등)는
      headless 캡처(③)로 즉시 채울 수 있는지 먼저 확인**했는가
- [ ] inventory.md의 모든 항목이 `완료` 또는 `불필요`인가
- [ ] 산출 파일이 규약 위치(프로젝트 루트 또는 사용자 지정)에 있고 경로를 안내했는가
- [ ] **산출물 규격 검증(pptx)**: `python scripts/verify_pptx.py <문서.pptx> --draft
      <원고>` — 생성 경로(번들/전용 skill/자작)와 무관하게 테두리·비율 왜곡·폭
      균일·캡션·마크다운 잔재·이미지 소실을 기계 검증. **② 서식 복제 경로는 필수**
      (번들 빌더는 self_check 로 대부분 자체 수행하지만 이중 확인으로 권장)
- [ ] **렌더 검수(pptx)**: `scripts/export_slides.ps1 -Path <문서.pptx>` 로 슬라이드를
      PNG 렌더해(PowerPoint COM, 미설치 시 soffice PDF 폴백) 표지·목차·대표 화면
