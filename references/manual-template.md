# 매뉴얼 표준 목차·화면 문서화 포맷

국내 SI 관리자/사용자 매뉴얼 관행(화면 캡처 중심, 번호 배지 + 번호별 설명)을 따르는
표준 형식. manual-draft.md는 이 구조로 작성하고, docx/pptx 변환은 output-formats.md를 따른다.

## 1. 표준 목차

```
표지          시스템명 / "사용자용"·"관리자용" 구분 / 버전 / 연도(또는 작성일)
CONTENTS      장·절 번호 + 페이지 번호

01. 시스템 개요
    1.1 시스템 소개          (목적, 주요 기능 요약 — 3~5문장)
    1.2 회원체계(권한체계)    (구분/특징 표: 관리자·일반회원 등 롤별 권한)
    1.3 접속 환경            (권장 브라우저, 접속 URL — 선택)

02. 사이트 접속 및 로그인
    2.1 접속 및 로그인        (URL 접속 → 로그인 절차, 단계별 번호)
    2.2 로그아웃/비밀번호 변경 (있는 경우)

03. 화면 및 기능 설명        ← 매뉴얼의 본체
    3.1 <첫 번째 메뉴>
    3.2 <두 번째 메뉴>
        3.2.1 <하위 메뉴>    (메뉴 계층이 2단이면 3.x.y까지 사용)
        3.2.2 <하위 메뉴>
    ...

04. 부록 (선택)
    4.1 FAQ / 문제 해결
    4.2 용어집 / 문의처
```

- 장 번호는 `01.` 두 자리, 절은 `3.2`, 하위 절은 `3.2.1` 표기를 쓴다.
- 3장의 절 순서는 실제 시스템의 메뉴 노출 순서를 따른다 — 독자가 화면과 문서를
  나란히 놓고 찾기 때문이다.
- 대상 독자가 "관리자"면 관리자 기능만, "사용자"면 사용자 기능만 담는다. 둘 다면
  문서를 분리 산출하는 것을 기본으로 한다(권한이 달라 화면 자체가 다르므로).

## 2. 화면 단위 문서화 포맷

3장의 각 절(= inventory의 기능 단위 하나)은 아래 순서로 작성한다:

```markdown
### 3.2.1 관리자                                  ← 절 번호 + 메뉴 라벨

관리자 페이지에 접속할 수 있는 관리자 계정을 등록/수정/삭제로 관리하는
화면입니다.                                        ← 화면 개요 1~2문장

**접근 경로**: 시스템관리 > 회원관리 > 관리자      ← 메뉴 트리 그대로

![SCR-005](screenshots/SCR-005_admin-list_annotated.png)
[사진 5] 관리자 계정 목록                          ← 캡션: [사진 N] + 화면명

① 아이디, 회원명으로 검색할 수 있습니다.
② [등록] 클릭 시 관리자 계정 등록화면 [사진 6]으로 이동합니다.
③ [수정] 클릭 시 수정화면으로 이동하며, [삭제] 클릭 시 관리자 계정이
   삭제됩니다.                                     ← 배지 번호와 1:1 대응

※ 관리자 계정은 최소 한 개 이상 필요하여 webmaster 계정은 삭제할 수
없습니다.                                          ← 예외·주의는 ※로

**권한**: 시스템 관리자만 접근 가능                 ← 있는 경우만
```

- 번호별 설명(①②③)은 annotated 스크린샷의 배지 번호와 반드시 1:1로 대응시킨다.
  배지를 생략한 화면(예외)만 번호 없는 불릿 목록으로 쓴다.
- 화면에 자체 번호가 매겨진 섹션(1. 2. 3. 폼 단계 등)을 설명할 때는 **화면 표기 번호의
  오름차순**으로 서술하고, 번호 없는 요소는 화면 내 시각 순서(위→아래)에 끼워 넣는다 —
  독자가 화면과 대조하며 읽기 때문이다.
- 스크린샷에 보이지 않는 요소(스크롤 아래 등)를 설명할 때는 "(화면 아래로 스크롤 시)"
  같은 위치 단서를 붙인다.
- 등록/수정 화면이 이어지면 같은 절 안에서 [사진 N+1]로 이어 쓰고, 필수 입력 안내를
  붙인다: `※ * 표기는 필수 입력 항목으로 반드시 기입해야 합니다.`
- 관리자 기능이 사용자 화면에 반영되는 경우 그 연계를 한 문장으로 설명한다:
  "여기서 등록된 기업 정보는 사용자 페이지의 '사업 안내 > 투자기업 소개'에 노출됩니다."
- 절차(step-by-step)가 필요한 기능(신청, 발급 등)은 `1. → 2. → 3.` 번호 목록으로
  행동 단위로 쓴다. 한 단계에 한 행동만 담는다.
- **절 번호 계층**: 3단 번호(3.x.y)를 쓰는 경우 부모 절 3.x는 그룹 메뉴명을 제목으로
  반드시 둔다(본문 없이 제목만이라도) — 부모 없는 3.2.1은 목차 계층을 깨뜨린다.
  부모를 둘 수 없으면 2단(3.x)으로 평탄화한다.

## 3. 표기 규약

| 대상 | 표기 | 예 |
|---|---|---|
| 버튼·링크 | 대괄호 | [등록], [수정], [삭제], [엑셀다운로드] |
| 화면·메뉴 이름 | 작은따옴표 또는 그대로 | '사업 안내 > 투자기업 소개' |
| 필수 입력 | 별표 언급 | ※ * 표기는 필수 입력 항목입니다 |
| 주의·예외 | ※ 접두 | ※ 삭제 시 복구할 수 없습니다 |
| 스크린샷 캡션 | [사진 N] 화면명 | [사진 3] 공지사항 목록 |
| 입력 예시 | 따옴표 | "홍길동"과 같이 입력합니다 |

- 문체: 존댓말 평서형("~합니다", "~할 수 있습니다"). 명령형("~하시오")과
  기술 용어(API, DB, 파라미터, 렌더링)는 쓰지 않는다.
- **URL 라우트·개발 경로(/admin, /login 등)를 본문에 쓰지 않는다** — 그 정보는
  inventory.md 전용이다. 예외는 "1.x 접속 환경"의 접속 주소 안내뿐이다.
- 화면 라벨은 화면에 보이는 그대로 옮긴다. 임의로 번역하거나 다듬지 않는다 —
  독자는 문서의 단어를 화면에서 그대로 찾기 때문이다.

## 4. placeholder 규약 (스크린샷 미확보 화면)

```markdown
> [스크린샷 필요: SCR-012 사용자 등록 화면 — /admin/users/new]
```

- 위치는 실제 스크린샷이 들어갈 자리 그대로. 캡션 줄은 쓰지 않는다(사진 번호가
  밀리지 않도록 사진 번호는 확보된 이미지에만 부여한다 — 변환 시 빌더가 placeholder에
  `[사진 -] 화면명 (추후 삽입)` 캡션을 자동 부여해 레이아웃을 통일한다).
- 문서 변환 시 이 블록은 회색 안내 상자로 렌더되며 **SCR-ID와 화면명만 표기된다** —
  라우트·캡처 실패 사유 같은 작업 메모는 최종 독자에게 노출되면 안 되므로
  placeholder 블록이 아니라 inventory.md에만 기록한다.
- inventory의 캡처 상태는 `대기`로 유지한다 — 이후 라이브 접근이 가능해지면
  이 목록만 재실행해 채운다.

## 5. 소스 온리(B 모드) 특칙·미확정 표기

- 화면 라벨·문구를 코드에서 확정할 수 없으면 추측해 쓰되 `[확인 필요]`를 붙인다.
  추측을 표시 없이 쓰면 검수자가 오류를 찾을 수 없다.
- **미확정 표기는 `[확인 필요]` 단일 형식만 쓴다** — "(확정 전)", "(미정)", "TBD" 같은
  자유 서식을 쓰면 검수 단계의 grep 한 번으로 잡을 수 없기 때문이다.
- 동작 설명은 코드 근거(밸리데이션, 필수 어노테이션, 삭제 확인 로직)가 있는 것만
  단정형으로 쓰고, 근거 없는 것은 쓰지 않는다.
