# 출력 형식별 변환 명세 (docx / pptx)

manual-draft.md와 screenshots/를 최종 문서로 변환할 때의 요구 명세. 생성 경로:

1. **번들 빌더(기본)** — 어떤 환경(CLI/데스크톱)에서든 동일 구조를 보장하는 표준 경로.
   이 문서의 규격이 내장되어 있고, 원고 사전 검증(validate_draft)과 자체 검증
   (마크다운 잔재·항목 초과·배지 미사용)까지 수행한다. 생성기를 즉석에서 자작하지 않는다.
   ```
   python scripts/build_pptx.py --draft manual-work/manual-draft.md \
     --screenshots manual-work/screenshots --out ..\관리자매뉴얼_시스템명_20260714.pptx \
     [--title 시스템명] [--audience 관리자용] [--version 1.0] [--date 2026년 7월]
   python scripts/build_docx.py --draft ... --screenshots ... --out <프로젝트 루트>\...docx
   ```
   의존성: python-pptx / python-docx (미설치 시 종료 코드 2와 함께 설치 명령 안내 —
   강제 설치하지 않고 사용자에게 확인).
2. **전용 skill(특수 경로)** — 사용자가 **참고 템플릿 pptx를 제공해 서식 복제**가 필요한
   경우에만 `anthropic-skills:docx`/`pptx`를 호출한다(번들 빌더가 못 하는 유일한 작업).
   이 명세를 호출 요구사항으로 전달하고, 산출 후 검수 절의 기계 검증을 동일 수행한다.
3. **최후 폴백**: 의존성 설치도 불가하면 manual-draft.md를 최종 산출물로 확정하고
   pandoc 등 변환 방법을 안내한다.

## 공통

- 산출 파일명: `<독자영역명>매뉴얼_<시스템명>_<YYYYMMDD>.docx|pptx` — 독자 영역명을
  접두로 쓴다(예: `사용자매뉴얼_`, `관리자매뉴얼_`, `시스템관리자매뉴얼_`,
  `기관관리자매뉴얼_`). 독자 영역이 여러 개면 **영역마다 별도 파일**로 산출한다.
  **저장 위치는 대상 프로젝트 루트(또는 사용자 지정 경로)이며 manual-work/ 내부에
  두지 않는다** — 중간 산출물 공간은 .gitignore 대상이라 최종본이 함께 무시될 수
  있기 때문이다.
- 이미지는 `_annotated.png`가 있으면 그것을, 없으면 원본을 쓴다 (번들 빌더는 자동 우선).
- 사전 처리로 `python scripts/resize_images.py manual-work/screenshots --max-width 1400`을
  실행해 둔다 — 원본 해상도 그대로 넣으면 파일이 수십 MB로 커지기 때문이다.
- 원고의 인라인 마크업(`**`, 백틱)은 변환 시 실제 서식(굵게 등)으로 치환하거나 제거한다.
  변환 후 산출물 텍스트 추출본에 `**`, `![` 가 남아 있으면 안 된다.
- placeholder 블록은 회색 배경의 안내 상자로 렌더하되 **SCR-ID와 화면명만 표기**하고
  이미지 프레임과 동일한 크기·위치에 배치하며 `[사진 -] 화면명 (추후 삽입)` 캡션을
  동일 위치에 넣는다. 라우트·실패 사유 등 작업 메모는 넣지 않는다(inventory 전용).
  조용히 누락시키지 않는다.

## docx 변환 명세

| 요소 | 명세 |
|---|---|
| 표지 | 시스템명(제목), "사용자 매뉴얼"/"관리자 매뉴얼", 버전, 작성일. 페이지 번호 없음 |
| 목차 | 자동 목차(TOC 필드) — 헤딩 1~3 수준까지 |
| 헤딩 계층 | Heading 1 = 장(01. 시스템 개요) / Heading 2 = 절(3.2 회원 관리) / Heading 3 = 하위 절(3.2.1 관리자) |
| 본문 글꼴 | 맑은 고딕 10~11pt (한국어 문서 표준) |
| 스크린샷 | 페이지 폭 맞춤(여백 제외 100%), 가운데 정렬. 이미지 아래 캡션 문단 "[사진 N] 화면명" 9pt 가운데 |
| 번호별 설명 | ①②③ 원문자 그대로 문단 목록으로 (자동 번호 목록 스타일로 바꾸지 않는다 — 배지와 문자가 일치해야 하므로) |
| ※ 주의 | 별도 문단, 굵게 또는 음영 처리 |
| 표 | 머리행 음영, 테두리 실선 (회원체계 표 등) |
| 머리글/바닥글 | 머리글: 시스템명 · 문서명 / 바닥글: 가운데 페이지 번호 |
| 페이지 | A4 세로, 여백 상하 2cm 좌우 2.5cm. 각 장(Heading 1)은 새 페이지에서 시작 |

## pptx 생성 명세

### 공통 규격

- 16:9 (13.333 x 7.5 in). 한국어 글꼴 맑은 고딕.
- 슬라이드 구성: ① 표지 → ② CONTENTS → ③ 장 간지(01/02/03...) → ④ 화면 슬라이드(본체) → 부록.
- 화면 슬라이드는 **슬라이드 1장 = 화면 1개(inventory 기능 단위 1개)**가 원칙.
  번호별 설명이 6개를 넘으면 같은 제목으로 슬라이드를 나눈다(제목에 (1)/(2) 표기).

### 화면 슬라이드 레이아웃 (신규 생성 시)

```
┌─────────────────────────────────────────────────┐
│ 03. 화면 및 기능 설명        3.2.1 관리자   (p)  │ ← 상단 헤더: 장 제목(좌) + 절 번호·라벨(우), 페이지 번호
├─────────────────────────────────────────────────┤
│ 화면 개요 1~2문장 (14pt)                          │
│                                                   │
│ ┌───────────────────────────┐  ① 설명 …          │
│ │                           │  ② 설명 …          │
│ │   스크린샷(annotated)      │  ③ 설명 …          │ ← 좌: 이미지(높이 우선 맞춤)
│ │                           │  ※ 주의 …          │    우: 번호별 설명(12pt)
│ └───────────────────────────┘                     │
│  [사진 N] 화면명 (9pt, 이미지 하단)                │
└─────────────────────────────────────────────────┘
```

- 이미지가 가로로 긴 화면(목록 위주)은 상(이미지)/하(설명) 배치로 전환한다.
- **스크린샷 고정 프레임**: 모든 화면 슬라이드에서 이미지는 **동일 위치·동일 크기의
  프레임**(세로형 좌측 / 가로형 상단, 시작 y 고정)에 비율 유지로 중앙 배치한다 —
  장표마다 이미지 크기·위치가 들쭉날쭉하면 문서 완성도가 급락하기 때문이다.
  캡션 위치도 프레임 하단에 고정한다. (번들 빌더 내장 — 전용 skill 경로에서도
  이 요구를 명시적으로 전달할 것)
- **스크린샷 테두리**: 모든 스크린샷에 검은색 실선 테두리(두께 기본값 0.75pt, 그림자
  없음)를 적용한다 — 흰 배경 캡처와 슬라이드의 경계를 살리기 위함이다.
  (번들 빌더 내장 — **템플릿 서식 복제 경로에서도 동일 적용**: 템플릿에 선례
  이미지가 없어도 이 규격을 따른다)
- 표지: 시스템명 크게, "관리자용/사용자용" 뱃지, 연도. 장 간지: 장 번호+제목만 크게.
- CONTENTS: **모든 장·절을 나열**하고 각 항목에 해당 슬라이드 번호를 표기한다.
  생성 마지막에 실제 번호로 갱신한다(분할 슬라이드는 첫 장 기준).
- **페이지 번호 정책: PPT 슬라이드 순번으로 고정한다** (표지 제외 전 슬라이드 우하단).
  콘텐츠만 세는 별도 연번은 목차 참조와 어긋나므로 쓰지 않는다.
- 본문 텍스트 상자는 왼쪽 정렬을 기본으로 한다(양쪽 정렬은 문장부호 고아 줄을 만든다).

### 참고 템플릿 pptx가 제공된 경우 (서식 복제)

조직의 기존 매뉴얼과 형식을 일치시키는 것이 목적이므로, 새 레이아웃을 만들지 말고
템플릿의 서식을 그대로 복제한다:

1. pptx skill의 편집 워크플로로 템플릿을 unpack한다.
2. 슬라이드 유형을 식별한다: 표지 / CONTENTS / 장 간지 / **완성도 높은 화면 슬라이드 1장**(서식 원본).
3. 화면 슬라이드 서식 원본을 화면 수만큼 복제하고, 텍스트(절 번호·라벨, 개요, 번호별
   설명, 캡션)와 이미지(스크린샷)만 교체한다. 번호 배지가 템플릿에 도형으로 있으면
   그 도형을 재배치해 쓰고, annotate_screenshot.py 합성은 생략한다(이중 배지 방지).
   삽입하는 모든 스크린샷에 위 규격의 검은색 실선 테두리를 적용한다(템플릿에 이미지
   선례가 없어도 생략 금지).
4. CONTENTS의 절 목록·페이지 번호를 실제 구성에 맞게 갱신한다.
5. 템플릿에 있던 예시 내용이 남지 않았는지 전수 확인한다(placeholder 텍스트 grep).

### 검수 (두 형식 공통) — 기계 검증 우선

텍스트 추출본으로 grep·카운트 검증을 먼저 수행한다 (번들 빌더는 일부를 자체 수행):

- **마크다운 잔재**: `**`, `![` 패턴 잔존 여부
- **미확정 마킹**: `확인 필요`, `확정 전`, `TBD`, `미정` 잔존 여부
- **pptx 분할**: 슬라이드별 설명 항목(•/①) 수를 세어 6개 초과가 없는지
- **목차 대조**: CONTENTS(또는 TOC)에 본문의 모든 절이 나열되고 번호가 실제 슬라이드
  번호(또는 페이지)와 일치하는지

이후 시각 검사: 이미지 누락/깨짐, 요소 겹침, 캡션-사진 번호 대응, 한글 깨짐,
문장부호 고아 줄. 전용 skill 경로면 그 skill의 QA 절차(렌더 후 육안 검사)를 수행한다.
